## 3.4 关键 Bug 攻克实录

### Bug 1：嵌套 DIV 内容截断

**现象**：正文只提取到了前半部分。

**根因**：`<div class="main">` 内部包含大量嵌套 `<div>` 元素，使用 `<div class="main">(.*?)</div>` 的非贪婪正则会在遇到第一个内部 `</div>` 时就停止匹配。

**修复**：将正则替换替换为逐字符深度计数器：

```python
depth = 0
pos = main_start
while pos < len(html_no_img):
    next_open = html_no_img.find('<div', pos + 1)
    next_close = html_no_img.find('</div>', pos + 1)
    if next_open != -1 and next_open < next_close:
        depth += 1          # 进入嵌套
        pos = next_open + 4
    else:
        if depth == 0:
            main_end = next_close  # 找到目标标签！
            break
        depth -= 1          # 退出嵌套
        pos = next_close + 6
```

### Bug 2：HTML 实体解码顺序错误（最致命 bug）

**现象**：转换后的文章里 5 张图片只剩 1 张，4 张图片的 Markdown 引用代码凭空消失。

**调试过程**：Claude 专门编写了调试脚本 `debug_conversion.py`，在每个转换步骤后追踪图片占位符的存活情况。发现图片在"final tag removal"步骤从 5 张骤减到 1 张。

**根因链条**：

1. 文本中有 `重量控制在 < 2 kg`，其中 `<` 在原始 HTML 中写作 `&lt;`
2. `html.unescape()` 将 `&lt;` 解码为 `<`，文本变为 `重量控制在 **< 2 kg**`
3. 后续 `<[^>]+>` 标签清理正则匹配到了 `< 2 kg` 并把 `>` 之后的远端字符全部视为"标签内部"
4. 虚假标签跨越数百字符，吞噬了中间 4 张图片的 Markdown 引用代码

**修复**：**将实体解码移到标签清理之后执行**。先清理 `<...>` 真实标签（此时 `&lt;` 还是无害的实体字符串），然后才执行 `html.unescape()` 安全解码。

> **核心教训**：HTML 实体解码必须在所有标签清理操作之后进行。`&lt;` → `<` 会产生虚假标签边界。

### Bug 3：KaTeX 公式丢失

**现象**：原文中的数学公式在转换后的 Markdown 中不见了。

**根因**：KaTeX 已将 LaTeX 渲染为纯展示性的 HTML 元素树（`<span class="katex-html">` 内部是 `mord`、`mbin` 等样式类），原始 LaTeX 源码看似已丢失。

**修复**：Claude 发现了 `<annotation encoding="application/x-tex">` 元素——KaTeX 在生成展示 HTML 的同时会将原始 LaTeX 保留在这里。通过正则提取这些 annotation 并包裹为 `$...$` 格式：

```python
body = re.sub(
    r'<annotation encoding="application/x-tex">(.*?)</annotation>',
    lambda m: f'${m.group(1)}$',
    body, flags=re.DOTALL
)
```

### Bug 4：HTML 语义元素映射不完整

**现象**：callout 提示框内部的标签（`<div class="lbl">`、`<strong>`）被错误处理。

**修复**：在 callout 替换函数中加入了内部标签的预处理，先转换 `<div class="lbl">` → 文本 + 冒号，再转换 `<strong>` → `**粗体**`，最后才清理剩余标签。
