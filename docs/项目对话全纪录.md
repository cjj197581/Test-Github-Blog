## 3.4 关键 Bug 攻克实录

### Bug 1：嵌套 DIV 内容截断

**现象**：正文只提取到了前半部分。

**根因**：`<div class="main">` 内部包含大量嵌套 `<div>` 元素，使用 `<div class="main">(.*?)</div>` 的非贪婪正则会在遇到第一个内部 `</div>` 时就停止匹配。

**修复**：将正则替换替换为逐字符深度计数器：

```python
depth = 0
pos = main_start
while pos < len(html_no_img):
    next_open = html_no_img.find('<div', pos + 1)
    next_close = html_no_img.find('</div>', pos + 1)
    if next_open != -1 and next_open < next_close:
        depth += 1          # 进入嵌套
        pos = next_open + 4
    else:
        if depth == 0:
            main_end = next_close  # 找到目标标签！
            break
        depth -= 1          # 退出嵌套
        pos = next_close + 6
```

### Bug 2：HTML 实体解码顺序错误（最致命 bug）

**现象**：转换后的文章里 5 张图片只剩 1 张，4 张图片的 Markdown 引用代码凭空消失。

**调试过程**：Claude 专门编写了调试脚本 `debug_conversion.py`，在每个转换步骤后追踪图片占位符的存活情况。发现图片在"final tag removal"步骤从 5 张骤减到 1 张。

**根因链条**：

1. 文本中有 `重量控制在 < 2 kg`，其中 `<` 在原始 HTML 中写作 `&lt;`
2. `html.unescape()` 将 `&lt;` 解码为 `<`，文本变为 `重量控制在 **< 2 kg**`
3. 后续 `<[^>]+>` 标签清理正则匹配到了 `< 2 kg` 并把 `>` 之后的远端字符全部视为"标签内部"
4. 虚假标签跨越数百字符，吞噬了中间 4 张图片的 Markdown 引用代码

**修复**：**将实体解码移到标签清理之后执行**。先清理 `<...>` 真实标签（此时 `&lt;` 还是无害的实体字符串），然后才执行 `html.unescape()` 安全解码。

> **核心教训**：HTML 实体解码必须在所有标签清理操作之后进行。`&lt;` → `<` 会产生虚假标签边界。

### Bug 3：KaTeX 公式丢失

**现象**：原文中的数学公式在转换后的 Markdown 中不见了。

**根因**：KaTeX 已将 LaTeX 渲染为纯展示性的 HTML 元素树（`<span class="katex-html">` 内部是 `mord`、`mbin` 等样式类），原始 LaTeX 源码看似已丢失。

**修复**：Claude 发现了 `<annotation encoding="application/x-tex">` 元素——KaTeX 在生成展示 HTML 的同时会将原始 LaTeX 保留在这里。通过正则提取这些 annotation 并包裹为 `$...$` 格式：

```python
body = re.sub(
    r'<annotation encoding="application/x-tex">(.*?)</annotation>',
    lambda m: f'${m.group(1)}$',
    body, flags=re.DOTALL
)
```

### Bug 4：HTML 语义元素映射不完整

**现象**：callout 提示框内部的标签（`<div class="lbl">`、`<strong>`）被错误处理。

**修复**：在 callout 替换函数中加入了内部标签的预处理，先转换 `<div class="lbl">` → 文本 + 冒号，再转换 `<strong>` → `**粗体**`，最后才清理剩余标签。

# 第七部分：HTML→Markdown 工具的工程化

## 7.1 用户需求

用户要求"将 HTML→Markdown 的转换逻辑写成 Python 脚本"——不是临时脚本，而是可复用的独立工具。

## 7.2 脚本设计

Claude 创建了 `tools/html2md.py`（约 260 行），具备以下特性：

**命令行接口**：
```bash
python tools/html2md.py <输入.html> <输出目录> [标题] [日期]
```

**11 步转换流水线**：

```
Step 1  读取源文件
Step 2  提取 base64 图片 → PNG 文件（先于正文处理，避免大字符串污染）
Step 3  深度计数器提取正文（初始 depth=1，精确匹配嵌套闭合标签）
Step 4  移除 <script> 和侧边栏导航
Step 5  HTML → Markdown 语义映射
        ├ 标题 h2~h4         → ## ~ ####
        ├ 段落/粗体/斜体/链接  → MD 等价
        ├ Callout 提示框      → > **LABEL:** ... （lbl div 预处理消除嵌套）
        ├ 版本标签            → *[标签]*
        ├ 表格                → GFM 管道表
        ├ KaTeX annotation    → $...$ / $$...$$
        ├ 列表/分割线/上标    → MD 等价
Step 6  剥离 div / span 外壳
Step 7  清理残留 table/tr/td 标签
Step 8  ★ 标签清理 → 实体解码（顺序不可颠倒！注释标注）
Step 9  移除 SVG 图注残留文字（可配置）
Step 10 空白行收束 + Figure 标题分离
Step 11 组装 Jekyll Front Matter + 输出 .md
```

**可配置项**（文件顶部常量）：
- `CALLOUT_LABELS` — callout 类型到 Markdown 标签的映射
- `STRAY_OVERLAY_LINES` — 需要从输出中剔除的 SVG 泄露文本
- `STRAY_FIG_CAPTIONS` — 需要移除的图片侧标注文本

## 7.4 审查发现的三个问题及修复

### 问题 1（🔴 严重）：Callout 嵌套 DIV 截断

**根因**：与教训 #2 完全相同的问题。Callout 内部有 `<div class="lbl">现状</div>`，正则 `r'<div class="callout\s+(\w+)"[^>]*?>(.*?)</div>'` 中的 `(.*?)` 会在遇到 `lbl` 的 `</div>` 时截断。

**修复前**：在 `_callout()` 函数内部试图处理 `<div class="lbl">`，但此时正则已截断，`lbl` 之后的内容已被丢弃。

**修复后**：在 callout 正则运行之前，先将整个 body 中的 `<div class="lbl">X</div>` 替换为 `**X：**`，消除嵌套结构：

```python
# 关键：先消除 callout 内部的嵌套 div（<div class="lbl">），
# 否则非贪婪 .*? 会被内部 </div> 截断（与教训 #2 相同的问题）
body = re.sub(
    r'<div class="lbl">(.*?)</div>',
    r'**\1：**', body, flags=re.DOTALL,
)
```

### 问题 2（🟡 中等）：Figure 标题未分离

**现象**：图片和 Figure 说明文字粘在同一行：
```
![Fig1](...png)Figure 5-1: Mie 级数精确 RCS vs 频率...
```

**修复**：新增 Step 10b，将 `![...](...)Figure X-Y: caption` 分离为：
```
![Fig1](...png)

*Figure 5-1: Mie 级数精确 RCS vs 频率...*
```

### 问题 3（🟢 低）：有序列表 `<ol>` 未处理

```python
body = re.sub(r'<ol[^>]*?>', '\n', body)
body = re.sub(r'</ol>', '\n', body)
```

## 9.5 工程经验总结

### ✅ 成功做法

1. **从最小可行产品开始**：先 hello-world 验证链路，再接入复杂内容
2. **分阶段提交**：每次逻辑独立的改动一个 commit，便于回滚和对比
3. **转换流程脚本化并工程化**：从一次性脚本演进为可配置、可复用的独立工具
4. **专用调试脚本**：逐步骤追踪数据流，精确定位 bug
5. **充分利用 GitHub 原生生态**：零 CI 配置，零维护成本
6. **代码审查对照经验库**：用之前记录的教训逐条审查新代码

### ⚠️ 重要教训

1. **HTML 实体解码必须在标签清理之后**——`&lt;` 变 `<` 产生虚假标签
2. **正则不适合解析嵌套结构**——用深度计数器逐字符扫描
3. **同一类 bug 会在不同地方复发**——Callout 的处理重犯了嵌套 DIV 的问题，说明需要建立"检查清单"机制
4. **KaTeX 渲染输出不可逆**——必须依赖 `<annotation>` 还原 LaTeX
5. **大文件分段处理**——先用 grep 了解结构，再分块读取
6. **HTTPS 被阻断时切 SSH**——公司网络环境下的 Git 推送备选方案
7. **Jekyll 文章必须同时满足两个条件**——`YYYY-MM-DD-` 文件名前缀 + YAML Front Matter
