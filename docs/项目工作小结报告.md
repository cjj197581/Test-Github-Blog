## 二、技术选型分析

### 2.1 平台选择：GitHub Pages

GitHub Pages 是 GitHub 提供的免费静态网站托管服务，具有以下优势：

- **零成本**：免费托管，无需购买服务器或域名
- **推送即部署**：代码推送到仓库后自动构建发布，无需手动 CI/CD 配置
- **版本控制**：所有文章纳入 Git 版本管理，历史可追溯
- **全球 CDN**：GitHub 自带 CDN 加速，访问速度快

### 2.2 静态站点生成器：Jekyll

Jekyll 是 GitHub Pages 原生支持的静态站点生成器：

- GitHub Pages 内置 Jekyll 构建环境，推送源码后自动编译
- Markdown 文件原生渲染，通过 Front Matter 管理元数据
- 主题生态成熟（Minima 为默认主题）
- 支持 Liquid 模板语法，可自定义布局

### 2.3 数学公式渲染：MathJax 3

MathJax 是业界标准的浏览器端 LaTeX 渲染引擎：

- 通过 CDN 引入，无需本地安装
- 支持 `$...$`（行内）和 `$$...$$`（块级）语法
- 与 Markdown 兼容良好，在代码块和纯文本中自动跳过

### 2.4 总体架构

```
用户编写 Markdown (.md)
        │
        ▼
   Git Push → GitHub
        │
        ▼
   GitHub Pages + Jekyll 自动构建
        │
        ▼
   静态 HTML + CSS + JS
        │
        ▼
   MathJax CDN (浏览器端渲染公式)
        │
        ▼
   最终呈现的博客页面
```

## 四、关键技术难点与解决方案

### 4.1 Base64 图片提取（挑战难度：★★☆☆☆）

**问题描述**：5 张技术图表以 base64 编码嵌入 HTML 的 `<img src="data:image/png;base64,...">` 标签中，每张图片的 base64 字符串长度在 170,000–325,000 字符之间。需要提取为独立文件并替换为 Markdown 图片引用。

**解决方案**：

```python
import re, base64

img_pattern = re.compile(r'<img src="data:image/(\w+);base64,([^"]+)"[^>]*>')

def replace_img(match):
    idx = len(img_map) + 1
    ext = match.group(1)    # "png"
    data = match.group(2)   # base64 string
    img_map[idx] = (ext, data)
    return f'![图{idx}]({{{{ "/assets/images/balloon-report-fig{idx}.{ext}" | relative_url }}}})'

html_no_img = img_pattern.sub(replace_img, html_src)

# Write extracted images
for idx, (ext, data) in img_map.items():
    with open(f"assets/images/balloon-report-fig{idx}.{ext}", "wb") as f:
        f.write(base64.b64decode(data))
```

**结果**：成功提取 5 张 PNG 图片，总大小约 827 KB，图片引用格式符合 Jekyll 的 `relative_url` 规范。

### 4.2 嵌套 DIV 内容提取（挑战难度：★★★☆☆）

**问题描述**：正文内容位于 `<div class="main">` 内，但该 div 包含大量嵌套子 div。使用正则 `<div class="main">(.*?)</div>` 的非贪婪匹配会在遇到第一个内部 `</div>` 时就停止，无法提取完整内容。

**错误的做法**：
```python
# 这样会过早截断——遇到第一个嵌套的 </div> 就停了
body = re.search(r'<div class="main">(.*?)</div>', html, re.DOTALL).group(1)
```

**正确的做法**：
```python
# 从 <div class="main"> 开始，维护嵌套深度计数器
main_start = html_no_img.find('<div class="main">') + len('<div class="main">')
depth = 0
pos = main_start
main_end = None

while pos < len(html_no_img):
    next_open = html_no_img.find('<div', pos + 1)
    next_close = html_no_img.find('</div>', pos + 1)

    if next_open != -1 and next_open < next_close:
        depth += 1          # 进入嵌套 div
        pos = next_open + 4
    else:
        if depth == 0:
            main_end = next_close  # 找到目标闭合标签
            break
        depth -= 1          # 退出嵌套 div
        pos = next_close + 6

body = html_no_img[main_start:main_end]
```

**原理**：遇到 `<div` 时深度 +1，遇到 `</div>` 时深度 -1。当深度回到 0 时，表示找到了与起始 div 匹配的闭合标签。

### 4.3 HTML 实体解码与标签清理的致命顺序 Bug（挑战难度：★★★★★）

**问题描述**：这是本次项目中最隐蔽、影响最大的 bug。经过数轮调试发现：

1. 文本中有 `重量控制在 **< 2 kg**`，其中 `<` 在原始 HTML 中是 `&lt;`
2. `html.unescape()` 将 `&lt;` 解码为 `<`，使文本变成 `重量控制在 **< 2 kg**`
3. 后续的 `<[^>]+>` 标签清理正则将 `**< 2 kg**` 整体视为一个"HTML 标签"而删除
4. 更严重的是，`<` 与某个远端的 `>` 配对，形成跨越数百字符的"假标签"，吞噬了 4 张图片的 Markdown 引用代码

**调试过程**：通过编写专用的调试脚本（`debug_conversion.py`），在每个转换步骤后追踪图片占位符的存活情况，精确定位到"After final tag removal"步骤时图片从 5 张骤减到 1 张。

**根因定位**：

```
步骤顺序（错误）：
  ① html.unescape()        ← 将 &lt; 解码为 <
  ② 标签清理 <[^>]+>       ← 假标签吞噬真实内容
  ③ 输出                    ← 内容残缺

步骤顺序（正确）：
  ① 标签清理 <[^>]+>       ← 在 &lt; 还是实体时清理真实标签
  ② html.unescape()        ← 安全地将 &lt; 解码为 <
  ③ 输出                    ← 内容完整
```

**核心教训**：

> **HTML 实体解码必须放在所有标签清理操作之后执行。** 这是因为实体解码会将无害的 `&lt;` 转变为有害的 `<`，从而打破"HTML 标签边界"的假设。

### 4.4 KaTeX 数学公式恢复（挑战难度：★★★★☆）

**问题描述**：原始 HTML 使用 KaTeX 渲染数学公式。KaTeX 的工作流程是先解析 LaTeX 源码，再生成纯展示性的 HTML 元素树。渲染后的 DOM 结构类似：

```html
<span class="katex">
  <span class="katex-mathml">
    <math>
      <semantics>
        <annotation encoding="application/x-tex">\sigma = \pi a^2</annotation>
      </semantics>
    </math>
  </span>
  <span class="katex-html">
    <span class="base">
      <span class="mord">σ</span>
      <span class="mrel">=</span>
      <span class="mord mathnormal">π</span>
      ...  <!-- 纯展示性元素，不可逆解析 -->
    </span>
  </span>
</span>
```

**关键发现**：KaTeX 在生成展示性 HTML 的同时，会将原始 LaTeX 源码保留在 `<annotation encoding="application/x-tex">` 元素中——这个 discovery 是公式能够成功恢复的关键。

**解决方案**：

```python
# 提取 KaTeX annotation 中的原始 LaTeX，转换为 MathJax 格式
body = re.sub(
    r'<annotation encoding="application/x-tex">(.*?)</annotation>',
    lambda m: f'${m.group(1)}$',
    body,
    flags=re.DOTALL
)
```

**限制**：如果 KaTeX 输出的 annotation 元素缺失（某些精简配置下可能不生成），则原始 LaTeX 源码不可恢复，需要手动重写公式。

### 4.5 HTML 语义元素到 Markdown 的映射（挑战难度：★★★☆☆）

原始 HTML 使用了丰富的语义元素，需要逐一映射为 Markdown 等价形式：

| HTML 元素 | Markdown 映射 | 说明 |
|-----------|--------------|------|
| `<h2>` ~ `<h4>` | `##` ~ `####` | 标题层级降一级（h1 用于 frontmatter） |
| `<div class="callout info">` | `> **ℹ️ INFO:** ...` | 提取 class 决定图标和标签 |
| `<div class="callout warn">` | `> **⚠️ WARN:** ...` | 警告类提示 |
| `<div class="callout danger">` | `> **🔴 DANGER:** ...` | 危险/关键提示 |
| `<div class="callout ok">` | `> **✅ OK:** ...` | 结论/建议 |
| `<table>` | GFM 管道表格 | 需处理表头分隔行 |
| `<span class="tag tag-new">` | `*[新增]*` | 版本标记 |
| `<span class="tag tag-expand">` | `*[大幅扩充]*` | 内容标记 |

Callout 转换的特殊处理——必须在转换前先处理内部的 `<div class="lbl">` 标签和 `<strong>` 标记：

```python
def callout_replacer(m):
    cls = m.group(1)
    inner = m.group(2)
    labels = {'info': 'ℹ️ INFO', 'warn': '⚠️ WARN',
              'danger': '🔴 DANGER', 'ok': '✅ OK'}
    label = labels.get(cls, cls.upper())
    # 保留 callout 内部的粗体等格式
    inner_text = inner
    inner_text = re.sub(r'<div class="lbl">(.*?)</div>', r'\1：', inner_text)
    inner_text = re.sub(r'<strong>(.*?)</strong>', r'**\1**', inner_text)
    inner_text = re.sub(r'<[^>]+>', '', inner_text)
    inner_text = re.sub(r'\s+', ' ', inner_text).strip()
    return f'\n> **{label}:** {inner_text}\n'
```

## 六、经验总结与教训

### 6.1 成功经验

1. **从最小可行产品开始**：先搭建能工作的 hello-world 博客（welcome 文章），验证 Jekyll + MathJax 的核心链路。确认公式渲染、图片加载、首页列表均正常工作后，再接入复杂的报告内容。这种"先验证再扩展"的策略避免了在复杂内容卡住时还要回退排查基础架构的尴尬。

2. **分阶段提交**：基础搭建和报告转换分成两次独立 commit。如果报告转换出现问题，可以精确回滚到基础搭建状态，而不影响已完成的博客骨架。

3. **转换流程脚本化**：将 HTML→Markdown 的转换逻辑写成 Python 脚本而非手动操作。好处：
   - 可重复执行（调试时反复运行，每次从头开始，状态干净）
   - 可逐步优化（每次修改一行逻辑重新运行即可看到效果）
   - 可复用（后续有类似的 HTML 文档转换需求时，修改输入路径即可）

4. **专用调试脚本**：遇到图片丢失的 bug 时，没有盲目猜测，而是写了 `debug_conversion.py` 在每个转换步骤后追踪图片占位符的存活状态，精确定位到"final tag removal"步骤。这种"逐步骤插桩调试"的思路是定位复杂数据流 bug 的最有效方法。

5. **充分利用 GitHub 原生生态**：完全依赖 GitHub Pages 内置的 Jekyll 构建流程，没有引入额外的 CI/CD 工具或第三方托管平台。GitHub 自动处理 SSL 证书、CDN 分发、Jekyll 构建等所有基础设施，零维护成本。

### 6.2 教训与避坑指南

1. **HTML 实体解码的时机**（最重要）：
   - ❌ 错误：先 `unescape()` 再清理标签 → `&lt;` 变 `<` 产生假标签
   - ✅ 正确：先清理标签再 `unescape()` → 真实标签先被移除，实体安全解码
   - 适用场景：任何涉及 HTML→纯文本转换的工程

2. **正则表达式的边界**：
   - 正则不适合解析嵌套结构（HTML/XML/JSON）
   - 对于嵌套匹配，使用深度计数器逐字符扫描
   - 正则可用于扁平结构的转换（如单层标签替换）

3. **渲染输出的不可逆性**：
   - KaTeX/MathJax 的 HTML 输出是纯展示性的，无法还原为 LaTeX
   - 转换时必须依赖原始 LaTeX 源码或 `<annotation>` 后备
   - 在设计自包含 HTML 文档时，应考虑保留原始数据源

4. **大文件的处理策略**：
   - 2.8MB 文件无法一次性传给 Read 工具（token 限制）
   - 先用 `grep` 提取文档结构（章节标题、图片位置）
   - 再分块读取关键区域
   - 转换过程中避免在内存中多次复制大字符串
