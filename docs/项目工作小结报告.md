**提交记录**：commit `8ff9a59` — "Add Windy research report post and HTML-to-Markdown conversion tool"

将第二阶段的一次性脚本重构为可复用的独立工具 **`tools/html2md.py`**（约 315 行）。

**命令行接口**：
```bash
python tools/html2md.py <输入.html> <输出目录> [标题] [日期]
```

**11 步流水线架构**：
```
Step 1  读取源文件
Step 2  提取 base64 图片 → PNG 文件（策略性前置，先于正文处理）
Step 3  深度计数器提取正文（初始 depth=1，严格匹配嵌套）
Step 4  移除 <script> 和侧边栏导航
Step 5  HTML → Markdown 语义映射
        ├ 标题 h2~h4         → ## ~ ####
        ├ Callout 提示框      → > **LABEL:** ...
        ├ 表格                → GFM 管道表
        ├ KaTeX annotation    → $...$ / $$...$$
        └ 列表/分割线/上标    → MD 等价
Step 6  剥离 div / span 外壳
Step 7  清理残留 table/tr/td
Step 8  ★ 标签清理 → 实体解码（顺序不可颠倒）
Step 9  移除 SVG 图注残留（可配置常量）
Step 10 空白行收束 + Figure 标题分离
Step 11 组装 Jekyll Front Matter + 输出 .md
```

**可配置项**（文件顶部常量，按项目修改）：
- `CALLOUT_LABELS` — callout CSS 类 → Markdown 标签映射
- `STRAY_OVERLAY_LINES` — SVG/侧边栏泄露文本列表
- `STRAY_FIG_CAPTIONS` — 图片侧标注文本列表

## 四、关键技术难点与解决方案

### 4.1 Base64 图片提取（难度：★★☆☆☆）

5 张技术图表以 base64 编码嵌入 HTML，每张 170K–325K 字符。使用 `re.compile + base64.b64decode` 提取并写入 PNG 文件，替换为 Jekyll `relative_url` 格式引用。

### 4.2 嵌套 DIV 内容提取（难度：★★★☆☆）

正则 `<div class="main">(.*?)</div>` 无法处理嵌套 div——`.*?` 在第一个内部 `</div>` 处就会停止。替换为逐字符深度计数器：遇 `<div` 则 `depth += 1`，遇 `</div>` 则 `depth -= 1`，`depth == 0` 时找到目标闭合标签。

### 4.3 HTML 实体解码顺序 Bug（难度：★★★★★，最关键）

**现象**：转换后 5 张图片只剩 1 张，4 张图片引用代码凭空消失。

**根因链条**：
1. 文本 `重量控制在 < 2 kg` 中的 `<` 原始为 `&lt;`
2. `html.unescape()` 将 `&lt;` 解码为 `<`
3. 后续 `<[^>]+>` 标签清理匹配到 `< 2 kg` 并跨越数百字符寻找 `>`
4. 虚假标签吞噬了中间 4 张图片的 Markdown 引用

**修复**：将实体解码移至标签清理之后。先清理 `<...>`（此时 `&lt;` 还是无害实体），再执行 `html.unescape()`。

> **核心教训：HTML 实体解码必须在所有标签清理操作之后执行。**

### 4.4 Callout 嵌套 DIV 截断（难度：★★★★☆ —— 教训复发）

**发现过程**：代码审查时逐条对照经验库，发现 Callout 处理重犯了嵌套 DIV 的问题。

**根因**：Callout `<div>` 内部有 `<div class="lbl">现状</div>`，正则 `r'<div class="callout\s+(\w+)"[^>]*?>(.*?)</div>'` 中的 `(.*?)` 在 lbl 的 `</div>` 处截断，之后的内容全部丢失。

**修复**：在 callout 正则运行前，先全局将 `<div class="lbl">X</div>` 替换为 `**X：**`，消除嵌套后再匹配外层。策略："先消除嵌套，再匹配外层"。

### 4.5 KaTeX 数学公式恢复（难度：★★★★☆）

KaTeX 渲染的 HTML 元素树（`mord`、`mbin` 等样式类）不可逆解析。关键发现：`<annotation encoding="application/x-tex">` 元素保留了原始 LaTeX 源码。通过正则提取这些 annotation 并包裹为 `$...$` / `$$...$$` 格式。

### 4.6 HTML 语义元素到 Markdown 的映射（难度：★★★☆☆）

| HTML 元素 | Markdown 映射 |
|-----------|--------------|
| `<h2>` ~ `<h4>` | `##` ~ `####` |
| `<div class="callout info">` | `> **ℹ️ INFO:** ...` |
| `<div class="callout warn">` | `> **⚠️ WARN:** ...` |
| `<div class="callout danger">` | `> **🔴 DANGER:** ...` |
| `<div class="callout ok">` | `> **✅ OK:** ...` |
| `<table>` | GFM 管道表格 |
| `<span class="tag tag-new">` | `*[新增]*` |

## 五、代码审查：经验库驱动的质量保障

### 5.1 审查方法

用户要求对 `tools/html2md.py` 进行审查，"结合之前在这个项目前期积累的经验教训，看看这些问题是不是都在该程序中体现了"。Claude 读取了 `memory/html-to-markdown-lessons.md` 中的四条核心教训，逐条对照验证。

### 5.2 审查结果

| # | 教训 | 审查结果 | 详情 |
|---|------|---------|------|
| 1 | 实体解码必须在标签清理之后 | ✅ 正确 | Step 8：第 236 行清理 + 第 238 行解码 |
| 2 | 正则不能解析嵌套 DIV | 🔴 发现同类 bug | Callout 处理中复发（已修复） |
| 3 | KaTeX 渲染不可逆 | ✅ 正确 | 从 annotation 提取 |
| 4 | Base64 先提取再处理 | ✅ 正确 | Step 2 在 Step 3 之前 |

### 5.3 关键发现

同一类 bug（嵌套 DIV 正则截断）在不同位置复发——正文提取（Step 3）已正确解决，但 Callout 处理（Step 5）又重犯了相同错误。这说明：**需要在代码中建立"检查清单"机制，对所有使用 `.*?` 匹配 HTML 结构的位置进行逐项排查。**

## 七、经验总结与教训

### 7.1 成功做法

1. **从最小可行产品开始**：先 hello-world 验证 Jekyll + MathJax 链路，再接入复杂内容
2. **分阶段提交**：4 次独立 commit，每次逻辑一致，便于回滚和审计
3. **转换流程脚本化 → 工具化**：从一次性脚本演进为可配置、可复用的 CLI 工具
4. **专用调试脚本**：`debug_conversion.py` 逐步骤追踪数据流，精确定位图片丢失 bug
5. **充分利用 GitHub 原生生态**：零 CI 配置，零维护成本
6. **经验库驱动的代码审查**：用记忆系统中的教训清单逐条审查新代码，发现同类复发 bug
7. **网络问题备选方案**：HTTPS 被阻断时快速切换 SSH

### 7.2 重要教训

| # | 教训 | 严重度 | 适用场景 |
|---|------|--------|---------|
| 1 | HTML 实体解码必须在标签清理之后——`&lt;` → `<` 产生假标签 | 🔴 致命 | 任何 HTML→纯文本转换 |
| 2 | 正则不适合解析嵌套结构——用深度计数器逐字符扫描 | 🔴 致命 | 解析 HTML/XML/JSON |
| 3 | 同一类 bug 会在不同位置复发——需要"检查清单"机制 | 🟡 中等 | 任何有多处类似逻辑的项目 |
| 4 | KaTeX 渲染输出不可逆——必须依赖 `<annotation>` 还原 LaTeX | 🟡 中等 | KaTeX→LaTeX 转换 |
| 5 | 大文件分段处理——先用 grep 了解结构，再分块读取 | 🟢 低 | 大数据文件处理 |
| 6 | HTTPS 被阻断时切 SSH——Git 推送备选方案 | 🟢 低 | 企业网络环境 |
| 7 | Jekyll 文章须同时满足文件名格式 + Front Matter 两个条件 | 🟢 低 | GitHub Pages 博客 |

